# Maintainer Playbook

*How this repository approaches the work. Written for the next person (or the next me) who has to pick this up cold.*

---

## Core Principles

### 1. Evidence over self-report

Nothing claimed in this repository is trusted because a human said so. The `evidence/` directory is the single source of truth for what has actually happened. If the evidence contradicts a claim in a README or a PR description, the evidence wins and the claim gets rewritten.

### 2. Machines regenerate; humans do not

`evidence/*.json` and `evidence/*.md` are never hand-edited. They are regenerated by `scripts/update-all-evidence.ps1`. This is a non-negotiable invariant. Hand-editing evidence breaks the audit property and undermines the whole project.

### 3. Config is the source of truth

Tracked repositories, PR numbers, SLA thresholds, and per-repo naming all live in `config/tracked-repositories.json`. Changes go there first. Scripts read config; config does not read scripts.

### 4. Fail loudly on structural drift

`scripts/validate-repo.ps1` checks that every piece of the system is where it is expected to be: config schema, script entrypoints, README markers, workflow files, schema contracts. CI runs this on every push. If validation fails, the PR does not merge.

### 5. Partial success is not failure

A six-hour refresh that failed to fetch data for one of seven repos is a partial success, not a total failure. The manifest records which steps succeeded and which did not. Downstream consumers (the dashboard, the README generator) honor partial success and surface the gaps explicitly rather than pretending they did not happen.

### 6. Reliability over features

A feature that fails silently is worse than no feature. When faced with a choice between "ship a partial feature and iterate" and "make the existing pipeline more trustworthy," pick reliability. The `ROADMAP.md` explicitly down-weights cosmetic changes and upstream marketing.

## Maintenance Decision Tree

### Is a package worth tracking?

Track a package if **all** of the following hold:

1. It has visible public signals of maintenance gap — "maintainers wanted" banner, pinned handoff issue, long-open PR queue with no recent maintainer engagement, abandoned release cadence relative to historical baseline.
2. It has measurable install volume — minimum 1,000 weekly npm downloads. Below this threshold, an abandoned package does not have enough downstream blast radius to justify the maintenance effort.
3. There is a specific remediable bug, test gap, or documentation issue with a clear fix — not "this package is vaguely not great."
4. The upstream repository is not archived or explicitly marked "do not use." If upstream has said "migrate elsewhere," that migration is itself the right answer, not this tracking effort.

If any of these fail, do not add the package to `config/tracked-repositories.json`. Tracking has a cost; don't pay it without a reason.

### Is a bug worth filing a PR for?

File a PR when:

- The bug is specific, reproducible, and has a minimal test case
- The fix is scoped — one concern per PR, no "while I'm in here" drive-by changes
- The fix preserves backward compatibility unless explicitly opted into a breaking change by the upstream maintainer in a prior thread
- The change is covered by a regression test or (for test-only PRs) the test covers a real behavior contract

Do not file PRs for:

- Style-only churn (whitespace, import ordering, renaming variables)
- Dependency bumps that the upstream maintainer has not asked for
- Typos in comments (these waste maintainer time without delivering user-visible value)
- "Modernization" that changes the public API without upstream buy-in

### Is an upstream maintainer ghosting me?

Set a realistic SLA expectation per repo in `config/tracked-repositories.json`. For unmaintained packages, the default 24h SLA is aspirational — the purpose is to generate evidence of how long review has been pending, not to pressure the upstream.

Escalation path when a PR has been open for 90+ days without response:

1. Leave a polite ping on the PR noting the time elapsed and offering to rebase / address any concerns
2. If no response after another 30 days, leave a comment documenting the non-response for downstream consumers
3. Do not fork or compete with upstream without an explicit handoff thread or archive signal from the upstream maintainer
4. Document the state in `docs/CASE_STUDIES.md`

The goal is never to embarrass or replace an upstream maintainer. The goal is to make the maintenance gap legible to downstream consumers who inherit the package's state.

## Operational Playbooks

### "The refresh pipeline is broken"

1. Check `https://github.com/dusan-maintains/oss-maintenance-log/actions` for the failing run
2. Look at `evidence/manifest.json` — the `run_status` field and per-step outcomes indicate which stage failed
3. If the failure is a GitHub API rate limit, check that `GITHUB_TOKEN` is set in the workflow — with a token the effective limit is 5000 req/hr; without, it is 60 req/hr
4. If the failure is a script parse error, run `./scripts/validate-repo.ps1` locally to reproduce
5. If the failure is a schema validation error, check `scripts/validate-evidence.js` output — the schema contracts in `schemas/` will show which field violated which constraint

### "The README is out of sync with the data"

1. The README sections inside marker comments (`<!-- TAGLINE:START -->`, etc.) are regenerated from evidence data — do not hand-edit them
2. Run `./scripts/update-all-evidence.ps1` locally to regenerate everything
3. If a section's content looks wrong, the root cause is in the data, not the generator — check the corresponding `evidence/*.json` file

### "I need to add a new tracked package"

1. Add a new entry to `config/tracked-repositories.json` with `owner`, `repo`, `package` (nullable for non-npm repos), `status_label`, `maintainer_needed_signal`, `tracked_pr_numbers`, and `review_sla_base_name`
2. Run `./scripts/validate-repo.ps1` to confirm the config parses
3. Run `./scripts/update-all-evidence.ps1` to generate the first snapshot
4. Commit both the config change and the generated evidence files in the same PR
5. Update `docs/CASE_STUDIES.md` with a new section explaining the package and the contribution

### "I need to remove a tracked package"

Rare — removals should be justified. Valid reasons:

- Upstream archived the repository
- Upstream published a "use this fork instead" notice
- The package was deprecated on npm with a replacement link
- You have merged enough PRs that the maintenance status no longer fits "abandoned"

To remove:

1. Delete the entry from `config/tracked-repositories.json`
2. Run `./scripts/update-all-evidence.ps1` — stale evidence files will not be cleaned up automatically; delete them manually if they need to go
3. Update `docs/CASE_STUDIES.md` with the removal reason and the final state at removal
4. Do not delete historical `status-*.json` snapshots — the 180-day window is part of the audit trail

### "I need to release a new version of `oss-health-scan`"

1. Bump `cli/package.json` version
2. Add an entry to `CHANGELOG.md` at the root, describing Added / Changed / Fixed / Removed
3. Ensure all `cli/test/*.test.js` still pass: `cd cli && npm test`
4. Create a GitHub Release tagged `cli-vX.Y.Z`
5. The `publish-cli.yml` workflow handles the npm publish step

## Commit and PR Hygiene

### Commit messages

- Use conventional commit prefixes (`feat:`, `fix:`, `chore:`, `docs:`, `test:`, `ci:`)
- Scope the commit to one concern — if a commit touches three unrelated files, split it
- Bot-generated commits use `chore(evidence):` prefix exclusively, reserved for automation

### PR descriptions

Every PR to this repository should state:

1. What the PR changes (one sentence)
2. Why the change is needed (one sentence to one paragraph)
3. How the change was validated (manual test, automated test, validation command)
4. What, if anything, downstream consumers should notice

PRs without validation steps are returned for revision. "It should work" is not validation.

## The Bot Account

`dusan-maintains-bot` is the automation identity that commits evidence refreshes. It is not a human. Its commits are:

- Always prefixed `chore(evidence):` with the ISO 8601 timestamp of the refresh
- Always signed via the GitHub Actions `GITHUB_TOKEN`
- Never modify files outside `evidence/` and the marker regions of `README.md`
- Never open issues or PRs on this repository — it can file SLA alert issues via `check-alerts.ps1`, but only when a tracked package crosses a critical threshold

Do not attribute bot commits as human contributions. The commit count on the repository graph is intentionally dominated by bot commits; this is the design.

## When To Escalate To A Human Decision

Some situations are not scriptable and require a human call:

- An upstream maintainer explicitly asks to be removed from tracking
- A tracked package gets a CVE filed and the fix requires a breaking change
- A downstream consumer opens an issue claiming the evidence here is inaccurate
- The repository is forked by an organization that wants to use it as infrastructure
- A reviewer or researcher requests the data under an academic or compliance framing

None of these have runbooks. Think it through, document the decision in an issue or discussion, and update this playbook with what you learned.
